1//! The Rust Imaging Library. A performant and high-level image processing crate for Rust.
2//!
3//! See benchmarks and more by viewing the [README](https://github.com/jay3332/ril#ril). It should
4//! also be noted that the README is updated more frequently than the documentation.
5//!
6//! Similarly, you can see the changelog [here](https://github.com/jay3332/ril/blob/main/CHANGELOG.md).
7//!
8//! # Installation
9//! The MSRV (Minimum Supported Rust Version) of this crate is **v1.61.0**.
10//!
11//! Add the following to your `Cargo.toml` dependencies:
12//! ```toml
13//! ril = { version = "0", features = ["all"] ]
14//! ```
15//!
16//! ## Installing from GitHub
17//! You can also use the unstable but latest version by installing from GitHub:
18//! ```toml
19//! ril = { git = "https://github.com/jay3332/ril", branch = "main", features = ["all"] }
20//! ```
21//!
22//! ## Using `cargo add`
23//! If you have cargo >= 1.62.0, you can use `cargo add ril --features=all`.
24//!
25//! ## Cargo Features
26//! RIL currently depends on a few dependencies for certain features - especially for various image encodings.
27//! By default RIL comes with no encoding dependencies but with the `text` and `resize` dependencies, which give you text
28//! and resizing capabilities respectively.
29//!
30//! You can use the `all` feature to enable all features, including encoding features. This enables the widest range of
31//! image format support, but adds a lot of dependencies you may not need.
32//!
33//! For every image encoding that requires a dependency, a corresponding feature can be enabled for it:
34//!
35//! | Encoding      | Feature | Dependencies                   | Default? |
36//! |---------------|---------|--------------------------------|----------|
37//! | PNG and APNG  | `png`   | `png`                          | no       |
38//! | JPEG          | `jpeg`  | `jpeg-decoder`, `jpeg-encoder` | no       |
39//! | GIF           | `gif`   | `gif`                          | no       |
40//! | WebP          | `webp`  | `libwebp-sys2`                 | no       |
41//!
42//! Other features:
43//!
44//! | Description                                               | Feature    | Dependencies        | Default? |
45//! |-----------------------------------------------------------|------------|---------------------|----------|
46//! | Font/Text Rendering                                       | `text`     | `fontdue`           | yes      |
47//! | Image Resizing                                            | `resize`   | `fast_image_resize` | yes      |
48//! | Color Quantization (using NeuQuant)                       | `quantize` | `color_quant`       | yes      |
49//! | Gradients                                                 | `gradient` | `colorgrad`         | yes      |
50//! | Enable all features,<br/> including all encoding features | `all`      |                     | no       |
51//!
52//! ### WebP Support limitations
53//! WebP support uses `libwebp`, which is a native library. This means that if you try to use the
54//! `webp` feature when compiling to a WebAssembly target, it might fail. We plan on making a
55//! pure-Rust port of `libwebp` in the future.
56//!
57//! For ease of use, the `all-pure` feature is provided, which is the equivalent of `all` minus the
58//! `webp` feature.
59//!
60//! # Getting Started
61//! Import the prelude which brings commonly used types and crucial traits into scope:
62//!
63//! ```no_run
64//! use ril::prelude::*;
65//! ```
66//!
67//! Because all errors from this crate are of the same type, ril provides a `Result` type
68//! which you can use in any function that leverages ril, such as the `main` function:
69//!
70//! ```no_run
71//! use ril::prelude::*;
72//!
73//! fn main() -> ril::Result<()> {
74//!     // code goes here...
75//!
76//!     Ok(())
77//! }
78//! ```
79//!
80//! Now you can use the `?` operator on anything that returns a ``Result`` for convenience.
81//!
82//! # Brief Guide
83//! A quick guide and overview of ril's interface.
84//!
85//! ## Opening an image
86//! The [`open`][Image::open] method should suit your needs:
87//!
88//! ```no_run
89//! # use ril::prelude::*;
90//! # fn main() -> ril::Result<()> {
91//! let image = Image::<Rgb>::open("my_image.png")?;
92//! # Ok(()) }
93//! ```
94//!
95//! The sole argument can be anything that implements [`AsRef<Path>`][AsRef], such as
96//! a string or a file.
97//!
98//! You might have also noticed that [`Image`] is generic with one type parameter,
99//! which can be anything that implements [`Pixel`]. It represents what type of pixel
100//! this image has - in this case, the image has RGB pixels.
101//!
102//! Common pixel formats are [`Rgb`] (colored) and [`Rgba`] (colored with transparency),
103//! which are found in the prelude. There are also grayscale counterparts, such as [`Luma`].
104//!
105//! ### Reading from a byte stream
106//! You can also read from raw bytes using [`from_bytes`][Image::decode_from_bytes]:
107//!
108//! ```ignore
109//! # use ril::prelude::*;
110//! # fn main() -> ril::Result<()> {
111//! let bytes = include_bytes!("image.png") as &[u8]; // Replace this with your own image data
112//! let image = Image::<Rgb>::from_bytes(ImageFormat::Png, bytes)?;
113//! # Ok(()) }
114//! ```
115//!
116//! The first argument is the encoding of the image, and the second is a slice of bytes, or anything
117//! that implements [`AsRef<[u8]>`].
118//!
119//! You can also use [`from_bytes_inferred`][Image::from_bytes_inferred] to
120//! infer the format from the byte slice without having to explicitly provide an encoding:
121//!
122//! ```no_run
123//! # use ril::prelude::*;
124//! # fn main() -> ril::Result<()> {
125//! # let bytes = [0].as_slice();
126//! let image = Image::<Rgb>::from_bytes_inferred(bytes)?;
127//! # Ok(()) }
128//! ```
129//!
130//! ## Creating your own image
131//! You can create your own image using the [`Image::new`][Image::new] method:
132//!
133//! ```no_run
134//! # use ril::prelude::*;
135//! let image = Image::new(256, 256, Rgb::new(255, 0, 0));
136//! ```
137//!
138//! The above creates a 256x256 RGB image with all pixels set to red.
139//!
140//! The first argument is the width of the image, the second is the height, and the third is the
141//! fill color. The pixel type of the image can be inferred from this argument, which is why we
142//! don't have to specify it explicitly as a type argument - Rust type inference is powerful and
143//! infers this for us.
144//!
145//! ### The `from_fn` method
146//! The [`from_fn`][Image::from_fn] method is a shortcut for creating an image from a function:
147//!
148//! ```no_run
149//! # use ril::prelude::*;
150//! let image = Image::from_fn(256, 256, |x, y| {
151//!     // Do something, maybe with `x` and `y`, and return a pixel
152//!     Rgb::new(x as u8, y as u8, 0)
153//! });
154//! ```
155//!
156//! The above is just an example. You specify the width, height, and the function that
157//! generates the pixels. It should take two parameters - `x` and `y`, which specify the position
158//! of the pixel to generate - and return a pixel.
159//!
160//! ## Encoding and saving images
161//! You can encode and save an image to a file with the [`save`][Image::save] method:
162//!
163//! ```no_run
164//! # use ril::prelude::*;
165//! # fn main() -> ril::Result<()> {
166//! // Here's the red image from before:
167//! let image = Image::new(256, 256, Rgb::new(255, 0, 0));
168//!
169//! image.save(ImageFormat::Png, "output.png")?;
170//! # Ok(()) }
171//! ```
172//!
173//! The first argument is the encoding of the image, and the second is the path to the file.
174//!
175//! You may have noticed this is a bit repetitive, and that it is possible to infer the encoding
176//! from the file extension. In cases like this, you can use the slightly slower
177//! [`save_inferred`][Image::save_inferred] method:
178//!
179//! ```no_run
180//! # use ril::prelude::*;
181//! # fn main() -> ril::Result<()> {
182//! # let image = Image::new(256, 256, Rgb::new(255, 0, 0));
183//! image.save_inferred("output.png")?;
184//! # Ok(()) }
185//! ```
186//!
187//! Now, you do not have to explicitly specify the encoding as it is inferred from the output path.
188//!
189//! ### Encoding and saving images to memory
190//! You can encode images to a memory buffer by using the [`encode`][Image::encode] method:
191//!
192//! ```no_run
193//! # use ril::prelude::*;
194//! # fn main() -> ril::Result<()> {
195//! # let image = Image::new(256, 256, Rgb::new(255, 0, 0));
196//! let mut out = Vec::new();
197//! image.encode(ImageFormat::Png, &mut out)?;
198//! # Ok(()) }
199//!
200//! // Do something with `out`
201//! ```
202//!
203//! The first argument is the encoding of the image, and the second is the output buffer that must
204//! implement [`Write`][std::io::Write].
205//!
206//! There is no filename to infer the encoding from, so in this case you have to explicitly
207//! specify the encoding.
208//!
209//! ## Manipulating images
210//! Now that you know how to create and save images, let's look at some of the ways we can modify
211//! them!
212//!
213//! ### Inverting images
214//! A common manipulation method would be inverting every pixel in the image. To do this, there are
215//! two methods which you can use:
216//!
217//! - [`invert`][Image::invert]: Inverts the image in-place
218//! - [`inverted`][Image::inverted]: Consumes the image and returns a new image with the inverted pixels
219//!
220//! A common pattern you'll see in this crate is that many methods have an in-place method and a
221//! not-in-place counterpart, in which the former can be useful for method chaining. One usually does
222//! not have any memory or performance benefits than the other.
223//!
224//! Anyhow, here's how you'd invert an image:
225//!
226//! ```no_run
227//! # use ril::prelude::*;
228//! # fn main() -> ril::Result<()> {
229//! let mut image = Image::new(256, 256, Rgb::new(255, 0, 0));
230//! image.invert();
231//! image.save_inferred("output.png")?;
232//! # Ok(()) }
233//! ```
234//!
235//! `(255, 0, 0)` (red) inverts to `(0, 255, 255)` (cyan), so that should be the color of the
236//! output image.
237//!
238//! We can also use the [`std::ops::Not`] trait to invert an image:
239//!
240//! ```no_run
241//! # use ril::prelude::*;
242//! # fn main() -> ril::Result<()> {
243//! let image = Image::new(256, 256, Rgb::new(255, 0, 0));
244//! (!image).save_inferred("output.png")?;
245//! # Ok(()) }
246//! ```
247//!
248//! Seems to be a bit cleaner than the first way, but it really just comes down to preference...
249//! and whether or not you have ownership of the image object; you likely want to stay away from
250//! cloning images for no benefit as it is a very expensive operation.
